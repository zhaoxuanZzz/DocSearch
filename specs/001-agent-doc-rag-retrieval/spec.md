# Feature Specification: Agent 文档检索与 RAG 增强系统

**Feature Branch**: `001-agent-doc-rag-retrieval`  
**Created**: 2026-03-17  
**Status**: Draft  
**Input**: User description: "我想设计一个agent文档检索系统，增强agent文档阅读，RAG召回的能力"

---

## Overview

本功能旨在构建一套面向 AI Agent 的文档检索与知识召回系统，使 Agent 在执行任务时能够准确、高效地从大规模文档库中检索相关内容，提升 RAG（检索增强生成）管道的召回率与精确率，同时增强 Agent 对长文档的深度阅读与理解能力。

---

## Clarifications

### Session 2026-03-17

- Q: PDF 文档的解析与分块策略 → A: PDF 须先经结构化文档转换器（Docling）转为 Markdown 再行分块，以保留表格等结构化内容的语义完整性
- Q: 检索结果是否需要定位到原文具体位置 → A: 需要精准定位，返回结果须携带页码 + 标题路径（Heading Breadcrumb）+ 元素类型（段落/表格/标题）+ 元素在该页内的顺序索引，使 Agent 可直接定位到原文
- Q: Agent 文档访问技能的范围 → A: Agent 具备三种文档技能：`query`（语义/关键词 RAG 召回）、`read`（按页码/章节顺序阅读文档）、`grep`（正则或关键词模式匹配检索），参考 Copilot 检索代码库的能力设计
- Q: Agent 是否需要自主决策检索策略 → A: 是，Agent 应根据目标范围内的文档数量、文档总体积等上下文因素，自主在 RAG 召回与直接阅读/grep 之间选择最优策略
- Q: Word/DOCX 文档的表格处理路径 → A: DOCX 也经 Docling 转为 Markdown 后再分块，与 PDF 走完全相同的统一管道
- Q: `read` 技能的翻页单位 → A: 同时支持固定 Token 数（可配置，默认 2000 token）和标题块（Heading Block）两种模式，调用方按需选择
- Q: 检索策略路由的决策主体 → A: 系统提供路由建议（含文档集指标 + 推荐策略 + 理由），Agent 读取后自主决定是否采纳；系统不强制执行路由，仅作顾问角色
- Q: 技术栈选型 → A: 后端 Python + DeepAgents；前端 React + Ant Design X；关系型数据库与向量库均使用 PostgreSQL（pgvector 扩展）；对象存储使用 MinIO；缓存使用 Redis

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Agent 精准召回文档片段 (Priority: P1)

当 Agent 执行任务需要领域知识时，能够向文档检索系统发出查询，系统返回与查询语义最相关的文档片段，供 Agent 作为上下文进行生成回答。

**Why this priority**: 这是整个系统的核心价值——Agent 需要准确的外部知识才能给出高质量回答，召回质量直接决定生成质量。

**Independent Test**: 可通过一组标准问答对（golden dataset）独立验证——给定查询后，验证返回的 Top-K 文档片段中是否包含正确答案所需的信息。

**Acceptance Scenarios**:

1. **Given** 文档库中存在与查询相关的内容片段，**When** Agent 发出自然语言查询，**Then** 系统在 Top-5 结果中返回包含答案的片段，召回率 ≥ 90%
2. **Given** 查询语义模糊或存在多种解读，**When** Agent 提交查询，**Then** 系统返回多角度相关片段，并附带相关性置信分数
3. **Given** 文档库中不存在相关内容，**When** Agent 查询，**Then** 系统明确返回"未找到相关内容"而非返回不相关片段

---

### User Story 2 - 长文档深度阅读与章节定位 (Priority: P2)

Agent 在处理超出其上下文窗口的长文档（如技术手册、研究报告）时，能够按需定位并读取文档的相关章节，而非一次性加载整个文档。

**Why this priority**: 长文档是真实业务中最常见的场景，上下文窗口限制是 Agent 文档理解的瓶颈，解决此问题可显著扩展系统可处理的文档规模。

**Independent Test**: 给定一份 100 页以上的 PDF 文档和一组针对不同章节的问题，验证系统能否准确定位并返回相关章节内容，无需人工提前分割文档。

**Acceptance Scenarios**:

1. **Given** Agent 上传一份长文档，**When** Agent 提问该文档某一特定主题，**Then** 系统返回文档中最相关的段落及其页码/位置引用
2. **Given** Agent 需要逐步深入阅读某章节，**When** Agent 请求展开某片段的上下文，**Then** 系统能返回该片段的前后相邻内容（上下文扩展）
3. **Given** 文档包含图表、表格等结构化内容，**When** Agent 查询相关数据，**Then** 系统能识别并返回对应的结构化内容描述

---

### User Story 3 - 文档库管理与增量更新 (Priority: P3)

文档管理员能够向系统批量上传、更新或删除文档，系统自动完成文档的解析、分块和索引构建，无需手动干预。

**Why this priority**: 文档库的持续更新是系统长期可用性的基础，支持增量更新避免全量重建索引的高昂成本。

**Independent Test**: 上传一批新文档后，无需重启系统，验证 Agent 能在 5 分钟内查询到新文档的内容；更新已有文档后，验证旧版本内容不再被检索到。

**Acceptance Scenarios**:

1. **Given** 管理员上传多种格式的文档（PDF、Word、Markdown、TXT），**When** 提交上传请求，**Then** 系统自动完成解析与索引，并在完成后通知管理员
2. **Given** 某文档内容已过时，**When** 管理员提交更新后的版本，**Then** 系统在索引更新完成后，检索结果反映最新内容
3. **Given** 某文档需要删除，**When** 管理员执行删除操作，**Then** 该文档的所有片段从检索结果中消失，不影响其他文档

---

### User Story 4 - 多路召回融合与重排序 (Priority: P2)

系统支持语义检索（向量相似度）与关键词检索（倒排索引）的混合召回策略，并通过重排序模型对结果进行二次精筛。

**Why this priority**: 单一检索策略在不同查询类型下均有盲区，混合召回 + 重排序是提升召回精度的关键路径。

**Independent Test**: 分别对包含专有名词的精确查询和模糊语义查询构建测试集，验证混合召回相比单一策略的 MRR（Mean Reciprocal Rank）提升 ≥ 15%。

**Acceptance Scenarios**:

1. **Given** 查询包含特定技术术语或产品型号，**When** Agent 查询，**Then** 系统通过关键词匹配确保精确术语的文档被纳入候选
2. **Given** 查询使用自然语言表述（如"如何解决登录失败的问题"），**When** Agent 查询，**Then** 系统通过语义向量检索找到内容相关但措辞不同的文档片段
3. **Given** 混合召回产生多个候选片段，**When** 系统进行重排序，**Then** 最终返回的 Top-3 片段与查询意图的相关性高于未排序前的 Top-3

---

### User Story 5 - Agent 三模式文档技能：query / read / grep (Priority: P1)

Agent 能够像 Copilot 检索代码库那样检索文档库：自然语言 `query` 获取语义相关片段、`read` 按章节顺序阅读文档内容、`grep` 在文档内进行模式/正则匹配。三种技能可针对单个或多个文档独立使用。

**Why this priority**: 三模式技能是 Agent 文档理解能力的基础接口，直接决定 Agent 能否灵活与文档交互；与 RAG 召回并列为最高优先级。

**Independent Test**: 准备一份技术手册，分别用三种技能提问：语义问题用 `query`、阅读第 3 章用 `read`、查找所有包含"API_KEY"字符串的位置用 `grep`，验证每种技能独立返回正确结果。

**Acceptance Scenarios**:

1. **Given** Agent 需要回答一个概念性问题，**When** Agent 调用 `query` 技能，**Then** 系统返回与问题语义最相关的 Top-K 文档片段，附带文档来源和精确位置信息
2. **Given** Agent 需要顺序阅读一份文档的某一章节，**When** Agent 调用 `read` 技能并指定页码或章节标题，**Then** 系统返回该位置起连续的文档内容，支持翻页/继续阅读
3. **Given** Agent 需要找到所有包含特定字符串或模式的位置（如错误码、配置项名称），**When** Agent 调用 `grep` 技能并指定模式，**Then** 系统返回所有匹配位置列表，包含文档来源、页码/段落号及匹配上下文各 1 行
4. **Given** Agent 对多个文档发起 `grep`，**When** 文档集包含 1~20 个文档，**Then** 系统在所有目标文档中完成匹配并按文档分组汇总结果

---

### User Story 6 - Agent 自主检索策略路由 (Priority: P2)

Agent 在收到用户任务后，根据目标范围内的文档数量、文档总体积以及查询类型，自主决定采用 RAG 召回还是直接阅读/grep 策略，无需用户手动指定检索方式。

**Why this priority**: 不同场景下最优策略差异显著——少量文档直接阅读更精准，大量文档必须依赖 RAG；自动路由让 Agent 在各场景下均表现最优，而无需用户理解底层检索机制。

**Independent Test**: 准备两个测试场景：（A）1 份文档 + 精确查找问题；（B）100 份文档 + 模糊语义问题。验证 Agent 在场景 A 自动选择 read/grep，在场景 B 自动选择 RAG 召回，且无需用户干预。

**Acceptance Scenarios**:

1. **Given** 目标文档集仅包含 ≤ 5 份文档（总体积较小），**When** Agent 收到查询任务，**Then** Agent 自动选择直接阅读（`read`）或 `grep` 策略，以获取更精准的定位结果
2. **Given** 目标文档集包含大量文档（数量超过阈值或总体积超限），**When** Agent 收到查询任务，**Then** Agent 自动选择 RAG 召回策略，通过向量检索快速缩小范围
3. **Given** Agent 采用自动路由完成检索，**When** 结果返回给用户，**Then** Agent 在回答中附带说明使用了哪种策略及原因（可观测性）
4. **Given** 自动路由选定 RAG 但结果置信度低（相关性分数低于阈值），**When** 文档总量允许，**Then** Agent 自动升级为直接阅读策略进行二次验证

---

### Edge Cases

- 查询文本超长（超过 512 个 token）时，系统如何截断或分段处理？
- 文档内容包含大量噪声（页眉页脚、广告、乱码）时，结构化转换后如何过滤？
- 多语言混合文档（中英文混排）的检索是否能同时覆盖两种语言的查询？
- 极短文档片段（少于 50 字）是否会影响向量质量？
- 文档索引未完成时，Agent 发起查询应如何响应？
- 并发大量 Agent 同时查询时，系统能否保持响应时间稳定？
- PDF 中的复杂表格（合并单元格、嵌套表格）经结构化转换后，是否能还原为完整可检索的 Markdown 表格？
- `grep` 技能对超大文档集（> 20 份）使用时，应返回明确错误并引导 Agent 缩小范围或改用 `query`
- 策略路由判断后，若两种策略性能差异微小（边界场景），应以哪种策略为准？（默认以 RAG 为准，避免过度读取大文档）
- 同一文档被多个 Agent 的 `read` 技能同时顺序阅读时，每个 Agent 的阅读状态独立，不互相干扰

## Requirements *(mandatory)*

### Functional Requirements

**文档摄入与处理**

- **FR-001**: 系统 MUST 支持以下文档格式的解析与入库：PDF、Word（.docx）、Markdown（.md）、纯文本（.txt）
- **FR-001a**: PDF 和 Word（.docx）文档 MUST 均经由结构化文档转换器（Docling）先转换为 Markdown 格式，保留表格、多栏布局、标题层级等结构语义，转换后的 Markdown 作为后续分块的唯一输入；Markdown 和 TXT 格式可直接进入分块流程
- **FR-002**: 系统 MUST 对文档进行智能分块（Chunking），每个分块保留完整语义单元（段落、小节、完整表格行），避免在表格行中间或句子中间截断
- **FR-002a**: 针对 Markdown 表格的分块策略 MUST 保证每个 Chunk 中表格内容完整：整张小表作为单一 Chunk；超长表格按完整行边界分割，且每个分割 Chunk 均保留表头行
- **FR-003**: 系统 MUST 为每个文档片段生成语义向量（Embedding），并存入向量索引
- **FR-004**: 系统 MUST 支持文档的增量更新 —— 新增文档无需重建全量索引
- **FR-005**: 系统 MUST 在文档索引完成后，提供可查询的状态反馈（成功/失败/处理中）

**检索与召回**

- **FR-006**: 系统 MUST 支持语义向量检索，实现基于语义相似度的 Top-K 文档片段召回
- **FR-007**: 系统 MUST 支持关键词全文检索，实现基于词频/BM25 的精确匹配召回
- **FR-008**: 系统 MUST 支持混合检索模式，融合语义检索与关键词检索的结果
- **FR-009**: 系统 MUST 提供重排序（Reranking）功能，对初步召回结果按与查询的相关性二次排序
- **FR-010**: 系统 MUST 在返回结果中为每个片段携带精准原文位置信息，包含：文档来源（文件名 + 唯一标识）、页码、标题路径（Heading Breadcrumb，如"第2章 > 2.3节 > 表2-1"）、元素类型（段落/表格/标题/列表项）、元素在当前页内的顺序索引、相关性分数；精准位置信息须足以使 Agent 通过 `read` 技能直接导航到该位置

**上下文管理与长文档阅读**

- **FR-011**: 系统 MUST 支持上下文窗口扩展 —— 返回目标片段的前后相邻内容（可配置扩展范围）
- **FR-012**: 系统 MUST 支持跨文档检索，单次查询可从多个文档中召回相关片段
- **FR-013**: 系统 MUST 对超长查询进行自动截断或分段处理，不因查询长度导致失败

**Agent 集成接口**

- **FR-014**: 系统 MUST 提供标准化的查询接口（支持自然语言输入），供 Agent 直接调用
- **FR-015**: 系统 MUST 支持批量查询，允许 Agent 在单次请求中提交多个查询
- **FR-016**: 系统 MUST 返回结构化结果，包含片段文本、元数据（来源、位置）、相关性分数

**管理功能**

- **FR-017**: 系统 MUST 支持文档的删除操作，删除后相关片段立即从检索结果中移除
- **FR-018**: 系统 MUST 提供文档库状态查询接口，展示已索引文档数量、片段数量、索引状态

**Agent 文档阅读技能**

- **FR-019**: 系统 MUST 提供 `query` 技能接口，接受自然语言查询 + 可选的目标文档列表，在指定文档集（为空则全库）内执行语义 + 关键词混合召回，返回 Top-K 片段及精准位置元数据
- **FR-020**: 系统 MUST 提供 `read` 技能接口，接受文档标识 + 起始位置（页码或标题路径），以以下两种模式之一返回内容（调用方按需选择）：（1）**固定 Token 模式**：每次返回不超过可配置 Token 上限的连续内容（默认 2000 token）；（2）**标题块模式**：每次返回一个完整的标题块（章节/小节）内容；两种模式均支持"继续阅读"翻页，返回下一段/下一节内容
- **FR-021**: 系统 MUST 提供 `grep` 技能接口，接受正则表达式或关键词模式 + 目标文档列表，在目标文档的完整文本中执行模式匹配，返回所有匹配位置（文档名、页码、标题路径、匹配行及前后各 1 行上下文）
- **FR-021a**: `grep` 技能 MUST 支持同时指定多个目标文档；当目标文档总数超过系统配置上限时，系统 MUST 拒绝请求并返回明确错误提示，引导 Agent 缩小范围或改用 `query`

**检索策略自主路由**

- **FR-022**: 系统 MUST 为 Agent 提供策略路由建议接口，输入为目标文档集描述（文档数量、总体积）和查询意图，输出为推荐的检索策略（`query`/`read`/`grep`）、支撑该推荐的文档集客观指标及理由；系统不强制执行路由，Agent 可自主决定是否采纳建议
- **FR-023**: 当目标文档集文档数量 ≤ 可配置阈值（默认 5）且总体积 ≤ 可配置上限（默认 1MB）时，路由决策 MUST 优先推荐 `read` 或 `grep`；超过阈值时 MUST 推荐 `query`（RAG 召回）
- **FR-024**: 当 `query` 召回的最高相关性分数低于最低置信阈值，且目标文档集体积允许直接阅读时，系统 MUST 自动建议 Agent 升级为 `read`/`grep` 进行二次验证
- **FR-025**: Agent 使用任意检索技能后，系统 MUST 在响应元数据中记录本次使用的策略类型及选择依据，用于可观测性与后续优化

### Key Entities

- **Document（文档）**: 用户上传的原始文件，包含文件名、格式、上传时间、索引状态等元数据
- **Chunk（文档片段）**: 文档经分块后的最小检索单元，包含文本内容、所属文档引用、精准位置元数据（页码、标题路径/Heading Breadcrumb、元素类型、元素顺序索引）、字符数；PDF 来源的 Chunk 额外保留其对应的 Markdown 原文行号范围
- **Embedding（向量）**: 文档片段的语义向量表示，与 Chunk 一一对应，存储于向量索引中
- **Query（查询）**: Agent 发出的检索请求，包含查询文本、检索模式（语义/关键词/混合）、Top-K 参数
- **RetrievalResult（检索结果）**: 系统返回的结果集，包含匹配的 Chunk 列表（每项含精准位置元数据）、各自的相关性分数、本次使用的检索策略类型
- **Index（索引）**: 系统维护的向量索引与倒排索引，支持高效相似度搜索与关键词匹配
- **RoutingContext（路由上下文）**: Agent 发起路由决策时提供的上下文信息，包含目标文档数量、总体积、查询意图类型；系统据此输出推荐策略与理由

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在标准问答评测集上，Top-5 召回率 ≥ 90%（即 90% 的问题的答案出现在返回的前 5 个片段中）
- **SC-002**: 混合召回策略相比单一语义召回，MRR（Mean Reciprocal Rank）提升 ≥ 15%
- **SC-003**: 单次查询响应时间（从提交到返回结果）在 95% 的情况下不超过 2 秒
- **SC-004**: 新文档上传后，索引完成并可被检索的时间不超过 5 分钟（针对 100MB 以内的文档）
- **SC-005**: 系统支持至少 10 万个文档片段的索引规模，且查询性能不明显下降
- **SC-006**: 文档内容更新后，旧版本内容在 10 分钟内从检索结果中消失
- **SC-007**: Agent 并发查询量达到 50 QPS 时，平均响应时间增长不超过 20%
- **SC-008**: 包含表格内容的问答在评测集上，经结构化转换（Docling）后的召回精度（Precision@5）相比直接 PDF 文本提取至少提升 20%
- **SC-009**: Agent 自主路由在评测集（含小文档集/大文档集/精确查询/语义查询场景）的策略选择准确率 ≥ 85%
- **SC-010**: 检索结果中的精准位置信息（页码 + 标题路径）能被 `read` 技能直接复用导航的比例 ≥ 98%（位置信息无效或无法导航的情况 < 2%）

## Assumptions

- 文档语言以中文和英文为主，系统需同时支持两种语言的查询与检索
- 单个文档大小不超过 200MB，单次批量上传不超过 50 个文档
- Agent 通过内部 API 调用文档检索系统，不需要终端用户直接交互的 UI 界面（管理员操作可通过管理界面完成）
- 向量模型的选择与更新由系统管理员负责，普通使用者无需关心底层向量模型细节
- 系统部署在内部私有环境，文档内容不会传输至外部第三方服务
- 分块策略默认按段落/小节进行，分块大小可由管理员配置，默认为 512 token 左右
- PDF 和 DOCX 文档在分块前须经结构化文档转换（Docling）将其转为 Markdown，Markdown 版本与原始文件均须持久化存储，以支持精准位置回溯和 `read`/`grep` 技能直接访问
- `grep`/`read` 技能同时操作的最大目标文档数量上限由管理员配置，默认为 20 份；超出上限的请求会被拒绝并返回明确提示
- 检索策略路由阈值（文档数量、体积）由管理员配置，默认阈值为初始推荐值，可根据实际性能测试结果调整

### Technology Stack

| 层次 | 技术选型 |
|------|----------|
| 后端服务 | Python + DeepAgents |
| 前端界面 | React + Ant Design X |
| 关系型数据库 & 向量库 | PostgreSQL（含 pgvector 扩展，统一存储文档元数据与向量索引） |
| 对象存储 | MinIO（存储原始文档文件及 Docling 转换后的 Markdown） |
| 缓存 | Redis（查询缓存、索引状态、路由建议缓存） |
| 文档结构化转换 | Docling（PDF / DOCX → Markdown） |

## Out of Scope

- 本系统不包含文档生成、摘要自动生成或内容创作功能
- 不包含用户权限管理与文档访问控制（假设所有 Agent 对文档库有统一访问权限）
- 不包含实时网页抓取或外部数据源的自动同步
- 不包含对话历史管理或多轮对话记忆（该能力属于 Agent 自身职责）
